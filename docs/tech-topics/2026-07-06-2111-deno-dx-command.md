# Deno 2.6、「dx」コマンドでnpxの代替を提供

## 概要

2026年、Denoはバージョン2.6をリリースし、npmおよびJSRパッケージに含まれるバイナリをインストールせずにその場で実行できる新コマンド「dx」を導入した。これはNode.jsエコシステムにおける`npx`と同等の役割を果たすツールであり、`deno x`として実装されている。アップグレード後に`deno x --install-alias`を一度実行することで、`dx`という短いエイリアスコマンドが利用可能になる。

## 背景

Node.jsのエコシステムでは、パッケージを恒久的にインストールせず一時的に実行する手段として`npx`が広く使われてきた。一方でDenoはセキュリティを重視した設計を採用しており、ファイルアクセスやネットワーク通信、環境変数の読み取りなどをデフォルトで制限するパーミッションモデルを持つ。今回の`dx`（`deno x`）は、この安全性のポリシーを保ったまま、npmおよびJSR双方のパッケージをワンコマンドで実行できるようにする狙いで追加された。あわせて、パッケージインストール時に実行される「ライフサイクルスクリプト（postinstallスクリプトなど）」を安全に管理するための`deno approve-scripts`コマンドや、依存関係の脆弱性をGitHubのCVEデータベースと照合してレポートする`deno audit`サブコマンドも同時に導入されている。

## なぜ重要なのか

サプライチェーン攻撃の多くは、パッケージインストール時に自動実行されるライフサイクルスクリプトや、CLIツールを一時実行する際に潜む悪意あるコードを起点としている。`dx`はダウンロードや実行の前に確認プロンプトを表示し、ライフサイクルスクリプトの実行にも明示的な承認を求めることで、`npx`で問題視されてきた「知らないうちに任意コードが実行される」リスクを構造的に低減する。また、`--ignore-read`や`--ignore-env`といったフラグにより、パーミッションの粒度をコマンド実行ごとに細かく調整できる点も、実務でのセキュリティ運用において重要な意味を持つ。

## 実務での活用例

- CIパイプラインで一時的にnpm/JSR製のCLIツール（コード生成ツールやリンターなど）を実行する際、`dx`を使うことでツールのインストール手順を省略しつつ、意図しないファイル・ネットワークアクセスを許可制限で防げる。
- 新しい依存パッケージを導入する前に`deno audit`を実行し、既知の脆弱性がないかを事前に確認するワークフローをチームの標準プロセスに組み込める。
- postinstallスクリプトを持つパッケージについて、`deno approve-scripts`で個別に承認・拒否を管理することで、依存関係を追加するたびにレビュー可能なセキュリティゲートを設けられる。

## 注意点

- `dx`はDeno 2.6以降でのみ利用可能であり、既存プロジェクトでは事前にDenoのアップグレードと`--install-alias`の実行が必要になる。
- npm由来のパッケージ実行においても内部的にはDenoのパーミッションモデルに従うため、Node.js環境と全く同じ挙動になるとは限らず、実行時に想定外の権限プロンプトが出るケースがある点に注意が必要である。
- `deno audit`が参照する脆弱性データベースの網羅性は継続的に発展途上であり、既存のnpm監査ツール（`npm audit`等）を完全に代替するものではない点も踏まえて併用を検討すべきである。
