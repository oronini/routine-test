# WASI 0.3 のネイティブ非同期(async)対応

## 概要

Bytecode Alliance は2026年6月、WASI(WebAssembly System Interface)の新バージョン「WASI 0.3」を正式リリースした。最大の目玉は、WebAssembly Component Model にネイティブの非同期処理(async)がファーストクラスの機能として組み込まれたことである。`async func`、`stream<T>`、`future<T>` といったプリミティブが追加され、これまで手作業でのワークアラウンドが必要だった非同期I/Oが、言語やランタイムを問わず統一的に扱えるようになった。

## 背景

WebAssembly はもともとブラウザ向けの高速な実行環境として設計されたが、WASI の登場によりサーバーサイドやエッジコンピューティングでも「サンドボックス化された汎用ランタイム」として活用が進んでいる。従来の WASI 0.2 では、ファイルI/Oやネットワーク通信のような非同期処理をコンポーネント間でやり取りする際、ポーリングやコールバックベースの回避策に頼らざるを得ず、言語ごとの実装差も大きな課題だった。WASI 0.3 は、この非同期処理の扱いを Component Model の仕様レベルで標準化することを目的に、2026年2月のプレビュー版を経て正式版として公開された。対応ランタイムとしては Wasmtime 43 以降や jco が挙げられている。

## なぜ重要なのか

非同期処理がコンポーネントモデルのネイティブ機能になったことで、異なる言語(Rust, Python, JavaScript など)で書かれた WebAssembly コンポーネント同士が、共通のI/Oプリミティブを介して効率よく連携できるようになる。これにより、並行処理を伴うサーバーサイドアプリケーションやマイクロサービス的なコンポーネント合成が、オーバーヘッドの少ない形で実現しやすくなる。Bytecode Alliance は WASI 0.3 を、長期サポートを見据えた「WASI 1.0」に向けた重要なマイルストーンと位置づけており、Component Model の非同期設計が今後の WebAssembly エコシステム全体の土台になっていく可能性が高い。

## 実務での活用例

- **エッジ関数のI/O効率化**: CDNエッジで動くWebAssemblyワークロードが、外部APIやKVストアへの呼び出しをブロッキングなしで処理し、リクエストあたりのレイテンシを削減する。
- **多言語コンポーネントの合成**: Rustで書かれた非同期処理コンポーネントと、Pythonで書かれたビジネスロジックコンポーネントを、同じ`stream<T>`/`future<T>`インターフェースで接続するプラグイン基盤の構築。
- **サーバーレスランタイムの高速化**: Wasmtimeなど対応ランタイム上で、複数のWASIコンポーネントが並行にI/Oを待ち合わせるパイプライン処理を実装し、スループットを向上させる。

## 注意点

- WASI 0.3 はまだ「WASI 1.0」のような長期安定版ではなく、0.3.x系のリリーストレインの中で後方互換を保ちながら継続的に改善されていく段階にある。本番導入時はランタイムのバージョン要件(Wasmtime 43以降など)を確認する必要がある。
- 非同期プリミティブが増えたことで、コンポーネント間のインターフェース設計やエラーハンドリングが複雑化する可能性があり、既存のWASI 0.2ベースの実装からの移行にはインターフェースの見直しが伴う。
- 各言語のツールチェイン(Rust, JavaScript向けjcoなど)の対応状況にはばらつきがあるため、採用前に自分たちが使う言語・ランタイムでのサポート状況を個別に確認することが望ましい。
