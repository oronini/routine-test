# Deno Desktop ― Webの資産をそのままネイティブデスクトップアプリへ

## 概要

Deno Desktopは、Deno 2.9（2026年6月25日リリース）で導入された実験的な`deno desktop`コマンドで、単一のTypeScriptファイルからNext.js・Astro・Deno Fresh・TanStack Start・Vite SSRといったフレームワークで書かれたWebプロジェクトまで、既存のWebコードをそのままクロスプラットフォームのネイティブデスクトップアプリケーションへ変換できる機能である。出力はDenoランタイムとWebレンダリングエンジンをまとめて1つにパッケージ化した、プラットフォームごとの再配布可能なバイナリとなる。

## 背景

これまでWeb技術でデスクトップアプリを作る選択肢としては、Chromium Embedded Framework（CEF）を丸ごとバンドルするElectronが事実上の標準だったが、その分アプリサイズが大きくなりがちという課題があった。TauriのようにOS標準のWebViewを利用してサイズを抑える後発ツールも普及してきたが、いずれもDenoエコシステムの外側にある別ツールとの組み合わせが必要だった。Deno Desktopは、Denoランタイム自身にデスクトップアプリのビルド機能を組み込むことで、追加のツールチェーンなしにWebプロジェクトをそのままネイティブアプリ化するという狙いで開発された。

## なぜ重要なのか

- **既存のWeb資産をそのまま転用できる**: Next.jsやAstroなど普段使っているフレームワークのコードを、大きな書き換えなしにデスクトップアプリの形で配布できる可能性がある。
- **既定でOS標準WebViewを使用**: CEFをバンドルせずOSネイティブのWebViewをデフォルトで利用するため、アプリサイズを大幅に削減できる。Denoの公式ドキュメントではWebView版で約40MB、CEF版で約150MBという数値が示されており、The Registerが実際にmacOS上でビルドして計測したところWebView版で約68.5MB、CEF版で約308.9MBだったと報告されている。
- **Electron/Tauriとの比較軸が増える**: ランタイムとビルドツールが一体化している分、既存のDenoプロジェクトからデスクトップアプリへの移行コストが低くなる可能性があり、デスクトップ展開の選択肢を検討するうえでの新たな比較対象になる。

## 実務での活用例

- 社内向けの管理画面やダッシュボードをNext.jsやAstroで開発している場合に、`deno desktop`コマンドで同じコードベースからオフライン利用可能なデスクトップ版クライアントを試作する。
- Electron製の既存デスクトップアプリについて、アプリサイズやメモリ使用量の削減を目的に、Deno Desktopへの移行が現実的かどうかを検証用プロトタイプで比較する。
- TanStack StartやVite SSRで構築したツール系Webアプリを、配布用にネイティブアプリ化してユーザーがブラウザを開かずに起動できるようにする。

## 注意点

- 2026年7月時点でも公式に「実験的（experimental）」と明記された機能であり、APIやコマンド仕様が今後変更される可能性が高いため、本番プロダクトへの採用は慎重に判断する必要がある。
- WebView版はOSごとにレンダリングエンジンの挙動差（Windows/macOS/Linuxでのレンダリング結果やAPIサポート状況の違い）が出やすく、Electronのように単一のChromiumで挙動を統一できるCEF版と比べて、クロスプラットフォームでの見た目・動作の一貫性検証がより重要になる。
- アプリサイズの数値はDeno公式とThe Registerの実測で差があり（WebView約40MB vs 約68.5MBなど）、計測条件によって結果が変わるため、自プロジェクトで実際にビルドして確認することが望ましい。
