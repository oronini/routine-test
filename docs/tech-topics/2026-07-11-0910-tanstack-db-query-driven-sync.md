# TanStack DBの「クエリ駆動型同期」がもたらすリアクティブなクライアントデータ層

## 概要

TanStackエコシステムの一部である「TanStack DB」は、クライアント側にリアクティブなコレクション（部分的なデータベースのようなもの）を持たせ、サブミリ秒レベルの差分クエリを実現するライブラリである。2026年に入り、ElectricSQLやPowerSyncといった同期エンジンとの連携が進み、「クエリ駆動型同期（Query-Driven Sync）」と呼ばれる仕組みが整備された。これは、あらかじめテーブル全体をローカルにダウンロードするのではなく、アプリが実行中のライブクエリの条件（述語）に応じて、必要な行だけをオンデマンドで同期する方式である。

## 背景

これまでのオフラインファースト・リアルタイムアプリ開発では、サーバーの状態をクライアントのキャッシュへ反映させる仕組みとして、SWRやTanStack Queryのようなフェッチ＆キャッシュ型のライブラリが主流だった。しかしこれらは基本的に「取得したデータをそのまま保持する」設計であり、複雑な絞り込みや結合、リアルタイム更新の反映は開発者が個別に実装する必要があった。TanStack DBは、ElectricSQLチームを中心に開発され、ローカルコレクションに対してSQLライクなライブクエリを発行できる点が特徴で、当初はテーブル全体をローカルに同期する方式が中心だった。2026年3月に発表された「クエリ駆動型同期」では、PowerSyncやTrailbaseといった同期バックエンドとの統合を通じて、ライブクエリの条件に基づき同期範囲を絞り込む「インクリメンタル同期」モデルへと発展した。

## なぜ重要なのか

モバイルアプリや大規模データセットを扱うWebアプリでは、テーブル全体をローカルに同期する方式はメモリやストレージを圧迫し、初期ロードにも時間がかかる。クエリ駆動型同期により、実際にUIが必要としている行だけがメモリに載るため、スケーラビリティが大きく向上する。また、複数の同期バックエンド（ElectricSQL、PowerSync、Trailbaseなど）を抽象化されたインターフェースの下で扱えることで、特定のベンダーへのロックインを避けつつ、リアルタイム同期とオフライン対応を両立できる点も意義が大きい。フェッチ＆キャッシュ型からリアクティブなローカルデータベース型へという設計思想の転換は、フロントエンドの状態管理のあり方そのものに影響を与えつつある。

## 実務での活用例

- 在庫管理や予約システムなど、大量データの中から現在表示中の範囲だけをリアルタイムに同期したいダッシュボードアプリ
- オフライン環境でも操作を継続でき、オンライン復帰時に自動でサーバーと整合を取りたいモバイルWebアプリ
- 複数ユーザーが同時に同じデータを編集する共同編集系アプリで、サーバー側の変更を即座にローカルのライブクエリへ反映させたい場合

## 注意点

- クエリ駆動型同期はまだ発展途上の機能であり、対応する同期バックエンド（ElectricSQL、PowerSync、Trailbaseなど）によって挙動やサポート範囲が異なるため、導入前に各バックエンドのドキュメントを確認する必要がある。
- ローカルにリアクティブなデータ層を持つ設計は、既存のTanStack Queryなどのフェッチ＆キャッシュ型の設計と混在させると状態管理が複雑化するおそれがあるため、アーキテクチャ全体での役割分担を明確にすることが望ましい。
- ライブクエリの述語に基づく同期範囲の決定はアプリのクエリ設計に強く依存するため、クエリ設計が適切でないと想定以上のデータが同期されメモリ使用量が増加する可能性がある。
