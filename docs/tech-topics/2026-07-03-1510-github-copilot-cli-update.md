# GitHub Copilot CLIの大型アップデートと従量課金制への移行

## 概要

2026年6月、MicrosoftはGitHub Copilotに関する大規模なアップデートを発表した。「Microsoft Build 2026」ではCopilot CLIに複数の新機能が追加され、`/experimental`モードでタブ切り替え可能なターミナルUI（TUI）が導入されたほか、複数のモデルファミリーで実装を相互批評させる「Rubber duck」機能が正式提供（GA）となった。また、音声認識をデバイス上で完結させる音声入力もGAとなり、定期的なプロンプト実行やバックグラウンドタスクを予約できるプロンプトスケジューリング機能も実験的に追加された。これと並行して、2026年6月1日からはCopilotの課金体系が「Premium Requests」方式から、入力・出力・キャッシュのトークン消費量に応じて課金される「GitHub AI Credits」方式へと移行した。

## 背景

GitHub Copilotはこれまで、月額プランの中でモデルごとに定められた「Premium Requests」の消費数によって利用量を制限する方式を採用していた。しかしモデルの多様化やエージェント的な利用（CLIによる自律的なコード編集・デバッグなど）が進むにつれ、リクエスト単位の課金では実際の計算コストとの乖離が大きくなっていた。これを受けてGitHubは2026年4月28日付の公式ブログで課金体系の見直しを予告し、6月1日付でトークン消費量に基づく従量課金制「GitHub AI Credits」へと移行した。月額のプラン料金自体は据え置かれたが、モデルの選択や利用量によって実質的なコストが変動するようになった。同時期にCopilot CLIも、単なるコマンド生成ツールから、Issue・PR・Gistへのアクセスや音声操作を備えたエージェント型の開発支援ツールへと機能を拡張している。

## なぜ重要なのか

課金モデルの変更は、開発者やチームがCopilotをどのモデル・どの頻度で使うかという判断に直接影響する。トークンベースの従量課金は、軽量なタスクでは安価になる一方、大規模なコンテキストを扱うエージェント的な利用では想定以上にコストが膨らむ可能性があり、利用状況の可視化とコスト管理がこれまで以上に重要になる。またCLIにRubber duck（複数モデルによる実装レビュー）やプロンプトスケジューリングといった機能が加わったことは、Copilotが単発の補完ツールから、継続的にコードベースを監視・改善する自律的なエージェントへと役割を広げつつあることを示している。これは他のAIコーディングツール（Claude Codeなど）とも共通する潮流であり、開発者はエージェントへの権限委譲とコストの両方を設計段階で考慮する必要が出てきている。

## 実務での活用例

- チームやプロジェクトごとにCopilotのトークン消費量を定期的に確認し、モデル選択（軽量モデルと高性能モデルの使い分け）によるコスト最適化を検討する。
- CLIの`/experimental`モードを検証環境で試し、Rubber duck機能による実装レビューや、プロンプトスケジューリングによる定型タスク（依存パッケージの棚卸し、ドキュメント同期など）の自動化を評価する。
- 音声入力機能について、ハンズフリーでの操作性やセキュリティ要件（音声データが端末外に出ない設計）を確認した上で、ペアプログラミングやレビュー作業での活用可否を検討する。
- 従量課金への移行に伴い、社内の利用ガイドラインや予算上限の設定を見直し、想定外のコスト増加を防ぐ運用ルールを整備する。

## 注意点

- トークンベースの課金は、プロンプトやコンテキストの与え方（リポジトリ全体を読み込ませるか、必要な範囲に絞るか）によってコストが大きく変動するため、利用者側の運用リテラシーが求められる。
- Rubber duckやプロンプトスケジューリングなど実験的機能は、正式提供前に仕様が変更される可能性があるため、本番運用への組み込みは慎重に検討する必要がある。
- 音声入力やCLIエージェントに広い権限を与える場合、誤操作や意図しないコード変更・実行を防ぐため、実行前の確認フローや権限スコープの制限を併用することが望ましい。
