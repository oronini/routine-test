# Remix 3、React を捨てて独自の全部入りフレームワークへ

## 概要

2026年4月30日、Remixチームは「Remix 3」のベータ版を公開した。最大の特徴は、これまでReact上のメタフレームワークとして開発されてきたRemixが、React依存を完全に廃止し、Preactをフォークした独自のコンポーネントモデルを採用した点にある。JSXの記法自体は引き続き利用できるものの、仮想DOM（Virtual DOM）は廃止され、`useState`のようなフックによる暗黙的な再レンダリングではなく、明示的な更新処理へと設計思想が転換された。あわせてRemix 3はFetch API上に構築されており、標準の`Request`・`Response`オブジェクトを用いることで、Node.jsだけでなくDeno・Bunなど、Fetch APIを実装する任意のランタイムで動作する。

## 背景

Remixはもともと2021年にReact Router開発チームによって生み出され、React上のフルスタックフレームワークとして人気を博した。しかし2025年5月26日、RemixチームはRemix v3において依存関係を最小限に抑え、スタックの大部分を自前で保有する方針を発表し、React本体を捨ててPreactのフォークへ移行することを表明した。同時に、Remix v2で培われたローダーやアクション、ルートベースのデータ取得といった機能群は、React Router v7に統合される形で引き継がれることも明らかにされた。これにより、既存のRemix v2ユーザーには「React Router v7へ移行する」か「Remix v3という新しいフレームワークへ乗り換える」かという二つの道が用意され、Remix v2からRemix v3への直接的な移行パスは提供されないという明確な決断が下された。

## なぜ重要なのか

この方針転換は、フレームワークが特定のUIライブラリのエコシステムに依存し続けることの限界を示す事例として注目されている。RemixチームはWeb標準API（Fetch、Request/Response）を基盤に据えることで、フロントエンドとバックエンドの間のコンテキストスイッチを減らし、ランタイム非依存性を高める狙いを明言している。また「Model-First Development」を掲げ、LLMがソースコードやドキュメント、ツールチェーンを理解しやすい設計を志向している点も、AIを前提とした開発体験を重視する2026年のフレームワーク設計の潮流を象徴している。特定フレームワークの持続可能性を評価する際に、UIライブラリへの依存度や移行コストがどのように設計判断へ影響するかを考える上での具体的な材料となる。

## 実務での活用例

- 既存のRemix v2プロジェクトを保守する場合、当面は互換性の高いReact Router v7への移行を優先し、Remix v3への切り替えは新規プロジェクトや実験的な用途に限定して検討する。
- Node.js以外のランタイム（Deno、Bunなど）でのエッジ実行を検討している場合、Remix v3のFetch API準拠設計が既存のNode専用実装よりも移植しやすい可能性があるため、PoC段階で動作検証を行う。
- 仮想DOMを介さない明示的な更新モデルを採用することで、大量のリスト描画やリアルタイム更新が多い画面のパフォーマンスチューニングに役立つか、ベータ版を用いて計測してみる。

## 注意点

- Remix 3は2026年7月時点でベータ版であり、本番環境での採用には安定版のリリースを待つのが望ましい。
- Remix v2からRemix v3への公式な移行パスは提供されていないため、既存資産（Reactコンポーネントやフック依存のロジック）をそのまま流用することはできない。
- Preactフォークという独自実装への依存は、Reactエコシステムの膨大なライブラリ資産（UIコンポーネント、状態管理ライブラリなど）が直接利用できなくなることを意味するため、既存のReact資産に強く依存するプロジェクトでは移行コストを慎重に見積もる必要がある。
