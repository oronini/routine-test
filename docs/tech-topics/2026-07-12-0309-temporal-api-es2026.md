# Temporal APIがES2026に正式採用 ― Node.js 26でデフォルト有効化

## 概要

JavaScriptの新しい日時操作API「Temporal」が、TC39でStage 4に到達しES2026仕様に正式採用された。ブラウザ側ではFirefox 139が2025年5月に先行実装し、Chrome 144が2026年1月に追随。サーバーサイドでもNode.js 26がTemporalをデフォルトで有効化してリリースされ、フラグなしで利用できる環境が急速に整いつつある。Test262には約4,500件ものテストが追加されており、TC39内では「ES2015以来最大の追加」と評されるほど規模の大きい変更となっている。

## 背景

JavaScript標準の`Date`オブジェクトは、タイムゾーンの扱いが曖昧で、ミュータブルな設計や紛らわしいAPI（月が0始まりなど）が長年の技術的負債となっていた。実務では`Moment.js`や`date-fns`、`Luxon`といったサードパーティライブラリで補ってきたが、言語標準としての解決策が求められていた。TC39のTemporal提案は2018年頃から議論が始まり、イミュータブルなオブジェクト設計、タイムゾーン・カレンダーの明示的な扱い、ISO 8601準拠の文字列表現などを軸に仕様策定が進められてきた。2026年に入り、主要ブラウザとNode.jsの双方で足並みが揃ったことで、実質的に「Baseline」入りが見えてきた段階にある。

## なぜ重要なのか

日時処理はWebアプリケーションのほぼすべての領域（予約システム、ログ集計、国際化対応UIなど）に関わるため、標準APIの刷新は影響範囲が非常に広い。Temporalはイミュータブルであるため、意図しない副作用によるバグを防ぎやすく、`Temporal.ZonedDateTime`や`Temporal.PlainDate`のように用途別の型を分けることで、タイムゾーン付き日時と純粋な日付を混同するミスを型レベルで防止できる。またNode.js 26でデフォルト有効化されたことで、フロントエンドとバックエンドの双方で同一のAPIを使って日時を扱えるようになり、サードパーティライブラリへの依存を段階的に減らせる可能性がある。

## 実務での活用例

- 新規プロジェクトでは日時比較や期間計算のロジックを、`Date`ではなく`Temporal.PlainDate`・`Temporal.ZonedDateTime`で設計し、タイムゾーンを跨ぐ処理での不具合を未然に防ぐ。
- 既存プロジェクトで`Moment.js`や`date-fns`に依存している箇所を洗い出し、Node.js 26以降・対応ブラウザをターゲットにできる範囲から段階的にTemporalへの置き換えを検討する。
- CI環境のNode.jsバージョンをまだ24系などに固定している場合、Temporalのデフォルト有効化を見据えてアップグレード計画に組み込む。
- Safari（Technology Preview）や一部ブラウザではまだ対応が完了していないため、フロントエンドでポリフィル（`@js-temporal/polyfill`など）の併用を検討する。

## 注意点

- Safariの正式対応はまだ完了しておらず、一部APIはフラグ付きでの提供にとどまっている。フロントエンドで即座に無条件採用するとブラウザ間の互換性問題が発生する可能性がある。
- Stage 4・ES2026採用はあくまで仕様上の確定であり、各ランタイム・ブラウザの実装が仕様変更前のバージョンに準拠している場合、細部の挙動差異が残っている可能性がある。移行前に対象環境の実装バージョンを必ず確認すること。
- `Date`からの移行は単純な置換では済まないケースが多く、特にタイムゾーンやカレンダーシステムを暗黙的に扱っていたコードは、Temporalの明示的な型設計に合わせた設計見直しが必要になる。
