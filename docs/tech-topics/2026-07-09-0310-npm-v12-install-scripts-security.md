# npm v12、サプライチェーン攻撃対策としてインストールスクリプトをデフォルト無効化

## 概要

2026年7月、npmのメジャーアップデートである「npm v12」がリリースされます。最大の変更点は、依存パッケージの`preinstall`・`install`・`postinstall`といったライフサイクルスクリプトが、`npm install`実行時にデフォルトで実行されなくなることです。あわせてGitリポジトリ経由や任意URL経由でのパッケージ取得もデフォルトで制限されます。スクリプトを実行したい場合は、`allowScripts`設定などを通じて明示的に許可リストへ追加する必要があり、これによりインストール時の自動コード実行が「デフォルト許可」から「明示的な許可制」へと転換されます。

## 背景

npmのインストールスクリプトは、ネイティブモジュールのビルドなど正当な目的で長年使われてきましたが、同時に「依存パッケージを追加するだけで任意コードが実行される」という、エコシステム最大級の攻撃対象領域(attack surface)にもなっていました。2026年に入ってからも、Axiosやモデルコンテキストプロトコル関連フレームワークを狙った国家関与とみられる攻撃キャンペーンや、`binding.gyp`ファイルを悪用してnode-gypのネイティブビルド経由で自己増殖するワーム型マルウェア（Shai-Hulud/Miasma系統）など、ライフサイクルスクリプトを起点とするサプライチェーン攻撃が繰り返し確認されてきました。npm v12はこうした一連のインシデントを踏まえ、GitHubが主導する形でエコシステム全体のデフォルトを安全側に倒す決定を下したものです。

## なぜ重要なのか

インストールスクリプトの自動実行は、開発者が`npm install`を叩いた瞬間に、CI/CD環境やローカル開発機で任意のコードがサイレントに実行されることを意味します。認証情報の窃取やビルド成果物への悪意あるコード混入は、この経路を突かれたケースが大半であり、フロントエンド・バックエンドを問わずnpmエコシステムに依存するほぼすべてのプロジェクトが潜在的な影響範囲に含まれます。デフォルト設定という「多くの開発者が意識せず従う既定値」を変えることは、個々のプロジェクトでの対策よりもエコシステム全体のリスクを底上げできるため、影響範囲が非常に大きい変更です。

## 実務での活用例

- npm 11.16.0以降へ先行アップグレードし、`npm approve-scripts --allow-scripts-pending`などで自プロジェクトの依存関係のうちどのパッケージがインストールスクリプトを必要としているかを事前に洗い出す。
- CI/CDパイプラインで、意図しないパッケージが新たにインストールスクリプトを要求していないか（サプライチェーン攻撃の予兆）を検知する仕組みを組み込む。
- ネイティブアドオン（node-gyp依存）を使うパッケージについては、`binding.gyp`が存在するだけで暗黙のビルドがブロックされる点を踏まえ、許可リストの棚卸しと定期的な見直しをルーティン化する。
- 社内向けのnpm利用ガイドラインを更新し、「スクリプト実行を許可する前にパッケージの出所とメンテナンス状況を確認する」というレビュー観点を明文化する。

## 注意点

- allowScriptsの許可リストはソースコード管理下に置かれレビュー対象となるため、運用ルールを整えないと「レビューなしに何でも許可する」形骸化したチェックになりかねません。
- npm以外にもYarn BerryやBunなど他のパッケージマネージャーも同様にライフサイクルスクリプトをデフォルトブロックする方向へ動いており、複数のパッケージマネージャーを併用しているプロジェクトでは設定差異に注意が必要です。
- 本記事の情報は執筆時点の公開情報に基づくものであり、正式リリース後に仕様や移行手順の詳細が変更される可能性があるため、npm公式のリリースノートを併せて確認してください。
