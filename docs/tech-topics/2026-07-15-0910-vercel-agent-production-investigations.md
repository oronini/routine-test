# Vercel Agent、ダッシュボードに統合され本番障害の「AI調査」機能がパブリックベータに

## 概要

Vercelは、これまでアラートのトリアージやプルリクエストのコードレビューを担っていた「Vercel Agent」を拡張し、ダッシュボードに常駐する形で提供を開始した。障害アラートが発火した際に自動で調査を開始する「Investigations」機能に加え、プロジェクトに関する質問へのチャット応答、承認ベースでの操作実行までを一つのエージェントでカバーする。Pro・Enterpriseチーム向けに段階的なロールアウトでパブリックベータが始まっている。

## 背景

これまでVercel Agentは、プルリクエストに対する自動コードレビューや、異常検知アラートの一次トリアージといった限定的な役割で提供されてきた。しかし本番運用で実際に問題が起きた際には、デプロイのログやランタイムエラー、コスト急増などの兆候を人間が横断的に確認し、原因を特定する作業が依然として必要だった。今回の拡張では、Vercel Agentがダッシュボード上の常設機能となり、アラート発火をトリガーに自動で調査を開始する「Investigations」や、承認を得た上で実際にアクションを実行する仕組みが追加され、障害対応のワークフローそのものに組み込まれる形になった。

## なぜ重要なのか

Investigationsは、失敗したデプロイやランタイムエラー、コスト急増といった事象を起点に、データのパターン変化や何が変わったかを自動的にトレースし、原因の推定と修正案の提示までを行う。PRレビュー機能では、Agentが生成したパッチを実際のビルド・テスト・リンターを使ったセキュアなサンドボックス内で検証し、検証済みの提案のみがPRに反映されるため、誤った修正案がそのまま提示されるリスクが抑えられている。また、権限が必要な操作を行う際にはスコープを限定した実行計画を提示し、人間の承認なしには変更が加えられない設計になっており、本番環境へのAIエージェントの関与を安全に広げる試みとして注目されている。

## 実務での活用例

- 本番でエラー率やレイテンシが急上昇した際に、Investigationsが自動的に該当デプロイやコミットの差分を洗い出し、一次調査のリードタイムを短縮する。
- コスト急増のアラートをトリガーに、原因となっている関数やルートをAgentに特定させ、対応の優先順位付けに活用する。
- プルリクエストごとにAgentのコードレビューを走らせ、実際のCI環境と同等のサンドボックスで検証済みの修正提案のみをレビュアーに提示させる。
- ダッシュボードのチャットからプロジェクトの構成やデプロイ履歴について質問し、オンコール対応時の状況把握を効率化する。

## 注意点

- パブリックベータ段階の機能であり、対象プランや段階的ロールアウトの状況によっては利用できない環境がある。本番の障害対応フローに組み込む前に対象範囲を確認する必要がある。
- Agentが提示する修正案やアクションはあくまで提案であり、権限が必要な操作には人間の承認が必須となる設計だが、承認フローの運用ルールをチーム内で明確にしておく必要がある。
- サンドボックスでの検証はビルド・テスト・リンターの範囲に限られるため、本番特有のデータや負荷条件に起因する問題まで完全に検出できるとは限らない点に留意する。
