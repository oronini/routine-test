# Svelteエコシステムの新展開 ― Mochi・pottz・Svelte TV

## 概要

2026年7月、Svelteコミュニティから新しいプロジェクトが相次いで登場した。SvelteKitの代替として島（アイランド）アーキテクチャとプログラマティックなルーティングを採用しBun上で動作する「Mochi」、SvelteKitアプリとBunランタイム、Webビューを1つの実行ファイルにまとめてネイティブデスクトップアプリとして配布できる「pottz」、そしてDOMの代わりにSvelteコンポーネントをWebGLへレンダリングしスマートTVや低メモリ環境のWebビュー向けに最適化する「Svelte TV」である。あわせて、Svelte 5.46では厳格なCSP（Content Security Policy）下でのインラインスクリプト実行を支えるnonce自動注入機能（`csp`オプション）が追加され、フレームワーク全体としてセキュリティ強化も進んでいる。

## 背景

近年のフロントエンド開発では、単一のメタフレームワークにすべてを任せるのではなく、用途に応じて異なるランタイムやレンダリング方式を選択する動きが広がっている。Svelteはコンパイラベースで軽量なランタイムを持つという特性上、Bunのような高速なJavaScriptランタイムや、ブラウザ以外のターゲット（デスクトップアプリ、スマートTV、組み込みWebビューなど)との親和性が高い。今回登場したMochi・pottz・Svelte TVは、いずれも公式のSvelteKitとは別の切り口で「Svelteの書き方はそのままに、実行環境を広げる」ことを目指すコミュニティ発のプロジェクトである。また、フレームワーク横断でXSS対策としてのCSP自動化が求められる流れの中で、Svelte本体にもnonce注入の仕組みが取り込まれた。

## なぜ重要なのか

- **配布形態の多様化**: pottzのように単一実行ファイルへバンドルできれば、Webの技術スタックのままElectron相当のデスクトップアプリを配布でき、開発チームは新たにネイティブ技術を学ぶコストを抑えられる。
- **非ブラウザターゲットへの対応**: Svelte TVのようにWebGLへの直接レンダリングを行うアプローチは、DOM操作が重いスマートTVや組み込みデバイスでも滑らかなUIを実現する可能性があり、Web技術の適用範囲を広げる。
- **パフォーマンスとシンプルさの両立**: Mochiが採用する島アーキテクチャは、必要な部分だけをハイドレートすることでペイロードを削減しつつ、SvelteKitに比べて薄いレイヤーでの実装を志向しており、パフォーマンスチューニングの選択肢を増やす。
- **セキュリティのデフォルト化**: `csp`オプションによるnonce自動注入は、これまで手動で管理する必要があったCSPの導入コストを下げ、XSS対策をより多くのプロジェクトで実践しやすくする。

## 実務での活用例

- 社内ツールやユーティリティアプリをpottzでバンドルし、Web技術のまま軽量なデスクトップ配布物として提供する。
- デジタルサイネージやスマートTV向けアプリをSvelte TVで構築し、通常のDOMベースSvelteアプリと同じコンポーネントを流用しつつWebGLレンダリングの恩恵を受ける。
- 高速な初期表示が求められるコンテンツサイトで、MochiのBun＋島アーキテクチャ構成を試験的に導入し、SvelteKitとのパフォーマンス比較を行う。
- 既存のSvelteKitプロジェクトでSvelte 5.46へアップデートし、`csp`オプションを有効化してインラインスクリプトのnonce管理を自動化、CSPポリシーの導入・運用負荷を下げる。

## 注意点

- Mochi・pottz・Svelte TVはいずれもコミュニティ発の新しいプロジェクトであり、公式のSvelteKitほどの実績やエコシステム（アダプター、ドキュメント、プラグイン）はまだ整っていないため、本番導入前にメンテナンス状況や更新頻度を確認する必要がある。
- pottzで配布するデスクトップアプリは、Electron等の既存ソリューションと比較してセキュリティモデルやアップデート配信の仕組みが異なる可能性があるため、配布前に権限管理や自動更新の設計を十分に検証すべきである。
- Svelte TVのようなWebGLベースのレンダリングは、アクセシビリティ（スクリーンリーダー対応など）やテキスト選択といった通常のDOMが提供する機能を失う場合があるため、対象デバイス・用途を見極めて採用を判断する必要がある。
- CSPのnonce自動注入機能を使う場合も、サードパーティスクリプトや外部リソースの許可設定は別途見直す必要があり、機能を有効化しただけで全てのXSSリスクが解消されるわけではない。
