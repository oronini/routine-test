# VS CodeのGitHub Copilotブラウザツールが正式提供、エージェントが実ブラウザを操作可能に

## 概要

GitHubは2026年7月1日、Visual Studio Code上のGitHub Copilotエージェントが実際のブラウザを操作できる「ブラウザツール」機能を正式提供（GA）とした。有料のCopilotサブスクリプション利用者にはデフォルトで有効化され、エージェントはページ遷移やクリック・入力・ホバー・ドラッグ操作に加え、ページ内容の読み取り、コンソールエラーの取得、スクリーンショットの撮影までを自律的に行えるようになった。同時期の6月アップデートでは、複数のエージェントセッションを並行実行できる機能や、AnthropicおよびOpenAIのモデルで100万トークンのコンテキストウィンドウをサポートする変更、セッションの端末間同期、クレジット消費の可視化機能なども導入された。

## 背景

これまでVS Code上のCopilotエージェントは、コード生成やターミナル操作は自律的に行える一方で、実装した変更が実際のWebアプリ上でどう動くかを確認する手段が限られていた。開発者は自分でブラウザを開いて動作確認し、その結果をチャットに書き戻すという手作業が必要だった。ブラウザツールのGA化は、この「実装とその動作確認」のループをエージェント側に閉じ込めることを狙ったものであり、あわせて導入された並行セッション機能や大規模コンテキストウィンドウ対応と合わせて、複数のタスクを同時に、かつ長いコンテキストを保持したまま自律的に進められる開発体験への布石となっている。

## なぜ重要なのか

エージェントが自ら画面を操作して結果を検証できるようになると、「コードは書いたが動作未確認」という状態を減らせる可能性がある一方で、権限管理と信頼性の両面で新たな論点が生じる。GitHubはセキュリティ面の設計として、エージェントが開いたタブにはCookieなどの認証情報を引き継がせない仕様を採用しており、開発者が明示的に認証済みタブを共有しない限り、ログインが必要な機能をエージェント単独ではテストできないようになっている。また、タスクを完了と判断する基準（Autopilotの権限レベル）も見直されており、エージェントにどこまで承認なしで作業を進めさせるかというガバナンス設計が、チーム導入における実務上の焦点になる。

## 実務での活用例

- UI実装後にエージェント自身へページ遷移・クリック操作・スクリーンショット取得を指示し、レイアウト崩れやコンソールエラーの有無を一次チェックさせる。
- 並行セッション機能を使い、複数の独立したUIタスク（コンポーネント修正・E2Eテストの追従修正など）を同時に走らせて、ブラウザタブを相互に汚染させずに検証させる。
- ログインが必要な画面の検証では、認証済みタブを開発者が明示的に共有する運用ルールを設け、エージェントに不用意な認証情報を渡さない設計にする。
- Autopilotの権限レベルを見直し、どのタスクまでを承認なしで進めさせるかをチームのポリシーとして事前に定めておく。

## 注意点

エージェントが開いたブラウザタブは既定でCookieを引き継がないため、認証が必要なフローの検証には別途タブ共有などの追加設定が必要になる。また、ブラウザ操作による自動検証はあくまで一次確認であり、実際のE2Eテストやユーザーによる目視確認を代替するものではない点に留意する必要がある。並行セッションや大規模コンテキストウィンドウの利用はトークン消費量の増加にもつながりうるため、コスト管理の観点からも利用状況を定期的に確認することが望ましい。
