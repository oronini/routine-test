# Next.js 16.3: Turbopackの永続ビルドキャッシュとメモリ最適化

## 概要

Next.js 16.3では、Turbopackに「永続ビルドキャッシュ（persistent build cache）」が導入され、`next build` の実行時にディスク上のキャッシュを再利用できるようになりました。あわせてメモリ解放（メモリエビクション）の改善も行われ、長時間の開発サーバー稼働時のメモリ使用量が大幅に削減されています。Vercel自身のプロダクションサイトでの計測では、50ルートのコンパイル後のメモリ使用量が21.5GBから2GBへと約90%削減されたと報告されています。

## 背景

Turbopackは元々、開発サーバー（`next dev`）向けの高速バンドラーとして提供されてきましたが、`next build` によるプロダクションビルドでは毎回ゼロからコンパイルを行う必要がありました。特にCI環境では、ビルドのたびに全ファイルを再コンパイルするコストが無視できない問題となっていました。Next.js 16.3では、`turbopackFileSystemCacheForBuild` という設定フラグを有効にすることで、ビルド時にもファイルシステムキャッシュを利用できるようになり、CIがキャッシュ済みの `.next` ディレクトリを次回実行に引き継ぐといった運用が可能になりました。

## なぜ重要なのか

大規模なNext.jsアプリケーションでは、CIのビルド時間がデプロイのボトルネックになりがちです。永続ビルドキャッシュにより、変更のないモジュールの再コンパイルを避けられるため、CIパイプライン全体の実行時間短縮につながります。また、開発体験の面でも、メモリエビクションの改善によって長時間のdevセッションでもメモリリークのような挙動が抑えられ、大規模モノレポや多数のルートを持つプロジェクトでの開発が快適になります。これらの改善はTurbopackが単なる開発用ツールから、本番ビルドを支える基盤技術へと進化していることを示しています。

## 実務での活用例

- CI/CDパイプラインで `.next/cache` ディレクトリをジョブ間でキャッシュとして永続化し、`turbopackFileSystemCacheForBuild` を有効化することで、2回目以降のビルド時間を短縮する。
- 大規模なモノレポ構成のプロジェクトで、開発サーバーを長時間起動したままにするワークフロー（ペアプログラミングや長時間のデバッグセッションなど）において、メモリ使用量の増大を気にせず作業を続けられる。
- Rust製React Compilerのサポートや `import.meta.glob` など周辺機能とあわせて、ビルドパイプライン全体の刷新を検討するタイミングでの採用。

## 注意点

- 永続ビルドキャッシュはまだ新機能であり、設定フラグを明示的に有効化する必要があるため、既存プロジェクトへの導入時は挙動の変化を十分に検証すること。
- キャッシュが古い依存関係やビルド設定の変更を正しく検知できないケースがないか、CI導入時にはキャッシュの無効化条件を確認しておくことが望ましい。
- メモリエビクションの改善は開発用ファイルシステムキャッシュの有効化が前提となるため、旧バージョンからのアップグレード時は設定オプションのデフォルト値の変更点を確認する必要がある。
