# WebMCP: ブラウザ内AIエージェントにサイトの機能を「ツール」として公開する新標準

## 概要

Googleは、WebページがJavaScript関数やHTMLフォームをMCP（Model Context Protocol）互換の「ツール」としてブラウザ内のAIエージェントに公開できる新しいWeb標準「WebMCP」を発表した。Chrome 149でオリジントライアルが開始されており、サイト側が明示的にツールを定義することで、AIエージェントはピクセルやDOMを推測しながら操作する従来の方式に頼らず、構造化された入出力仕様に基づいて確実にサイトを操作できるようになる。Expedia・Booking.com・Shopify・Etsy・Instacart・Targetなどが早期導入企業として名を連ねている。

## 背景

これまでブラウザ内のAIエージェント（Gemini in Chromeなど）がWebサイトを操作する際は、ページのスクリーンショットやDOMツリーを解析し、ボタンやフォームの位置を推測してクリック操作をシミュレートする「視覚ベース」のアプローチが主流だった。この方式は不安定でレイテンシが大きく、開発者がCSSクラス名を変更したりUIをリファクタリングしたりするだけでエージェントの操作が壊れてしまうという課題があった。WebMCPは、AIアプリケーションと外部ツールを接続する標準プロトコルであるMCPの考え方をブラウザ・Web標準の領域に拡張したもので、サイト運営者が「エージェントに使わせたい機能」を明示的な契約として宣言できるようにする。

## なぜ重要なのか

WebMCPには命令的（Imperative）API と宣言的（Declarative）APIの2種類が用意されている。命令的APIでは、フォーム入力・ナビゲーション・状態管理などのツールを標準的なJavaScriptで自由に定義できる。宣言的APIでは、既存のHTMLフォームに注釈を追加するだけでWebMCPツールとして公開できる。いずれの場合も、サイトはWebMCPマニフェストを配信して明示的にオプトインする必要があり、ユーザーもエージェントへのアクセス許可を与える必要があるなど、双方の合意なしにはエージェントがツールを利用できない設計になっている。初期のサードパーティ計測では、視覚ベースのエージェント操作と比較してタスク完了速度が8〜12倍向上したという報告もあり、AIエージェントがWebサービスを「使う」という新しいユーザー体験の実用性を大きく引き上げる可能性がある。

## 実務での活用例

- ECサイトの「カートに追加」「注文確定」といった主要フローを宣言的APIでWebMCPツール化し、購入代行エージェントからの利用に対応する。
- 検索・フィルタリングなどの機能を命令的APIでツールとして公開し、入出力スキーマを明示することで、エージェントが誤操作しにくい構造化されたインターフェースを提供する。
- 既存のフォームベースのUIに段階的に注釈を追加し、DOM構造やCSSクラスをリファクタリングしてもエージェント連携が壊れないようにする。
- Chrome DevToolsのAI連携機能と組み合わせ、WebMCPツールの呼び出しログをデバッグしながらエージェント視点での挙動を検証する。

## 注意点

- オリジントライアル段階の実験的な仕様であり、正式な標準化やAPIの詳細仕様は今後変更される可能性がある。本番導入前に最新のドキュメントを確認する必要がある。
- ツールとして公開する機能は、悪意あるエージェントや誤動作するエージェントからの呼び出しも想定し、認可・レート制限・入力値検証などのサーバーサイドの安全対策を別途講じる必要がある。
- ユーザーの同意なしにエージェントがサイトを操作できないよう、ブラウザ側のアクセス許可モデルを正しく理解し、意図しない自動操作が行われないUI設計を心がける必要がある。
