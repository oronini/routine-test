# CodeQL 2.26.0：AIのシステムプロンプトインジェクションを検出する新クエリ

## 概要

2026年7月10日、GitHubはコード解析エンジンCodeQLの最新バージョン2.26.0をリリースした。Kotlin 2.4.0対応などに加え、今回の目玉はJavaScript/TypeScript向けに新設された`js/system-prompt-injection`クエリである。これは、信頼できないユーザー入力がAIモデルのシステムプロンプトに直接流れ込み、モデルの挙動を攻撃者が操作できてしまう脆弱性を静的解析で検出するものだ。あわせてOpenAI・Anthropic・Google GenAIなど主要なAI SDKに対するプロンプトインジェクションの「シンク（危険な流入先）」定義も拡充された。

## 背景

LLMを組み込んだアプリケーションが急速に普及する一方で、ユーザー入力や外部データがそのままシステムプロンプトやツール定義に混入し、モデルの指示を書き換えられてしまう「プロンプトインジェクション」は依然として代表的な脆弱性の一つである。これまでCodeQLのセキュリティクエリは、SQLインジェクションやコードインジェクションといった伝統的な脆弱性パターンを中心にカバーしてきたが、AI SDKの利用が一般化するにつれて、AI特有のデータフローに対応したクエリの必要性が高まっていた。2.26.0では、OpenAIのRealtimeセッション指示やSora向けプロンプト、Anthropicのレガシーcompletion API、Google GenAIのシステム指示やキャッシュコンテンツなど、具体的なSDK呼び出しを「危険な流入先」として新たに定義し、テイント解析の対象を広げている。

## なぜ重要なのか

システムプロンプトはAIモデルの動作方針や権限境界を規定する土台であり、そこに信頼できない値が混入すると、モデルに本来意図しない指示を実行させたり、機密情報を漏洩させたりする攻撃の起点になり得る。手動レビューだけでこの種のデータフローを見つけるのは難しく、特にAI SDKの呼び出し箇所が複数のファイルやモジュールにまたがる場合は見落としが生じやすい。CodeQLのようなデータフロー解析ベースの静的解析ツールにAI特有の脆弱性パターンが組み込まれたことで、CI/CDパイプラインの一部としてプロンプトインジェクションのリスクを継続的かつ自動的にチェックできるようになる点が重要である。

## 実務での活用例

- GitHub Code Scanningを有効化しているリポジトリで、CodeQL 2.26.0への自動更新後に`js/system-prompt-injection`のアラートが出ていないか確認する。
- 自社のAIチャットボットやエージェントの実装で、ユーザー入力やRAGで取得した外部コンテンツがシステムプロンプトやツール定義に直接連結されていないか、CodeQLの検出結果を起点に洗い出す。
- OpenAI・Anthropic・Google GenAIなど複数のSDKを併用しているプロジェクトで、新しく追加されたシンク定義をもとに、これまで見落とされていたAPI呼び出し箇所を重点的にレビューする。
- 検出された箇所については、入力のサニタイズやプロンプトテンプレートの分離（ユーザー入力を独立したメッセージロールに分ける等）といった対策を適用し、再スキャンで解消を確認する。

## 注意点

- 静的解析はコード上のデータフローを検出するものであり、実行時の挙動やモデル自体の頑健性までは保証しない。検出結果はあくまで注意喚起であり、実際の悪用可能性は個別に評価する必要がある。
- 新しいクエリはfalse positive（誤検知）を含む可能性があるため、アラートが出た箇所がすべて実害のある脆弱性とは限らない点に留意する。
- 本クエリはJavaScript/TypeScript向けであり、他言語でのAI SDK利用については対応状況が異なる場合があるため、自社の技術スタックでの対応言語を事前に確認する必要がある。
